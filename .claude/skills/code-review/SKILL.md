# code-review — マルチペルソナ・コードレビュースキル

## 目的

現在の作業ブランチ（`main` との差分）を、複数の専門ペルソナの視点から並列にレビューし、仕様との整合性・規約遵守・コード品質を多角的に検証して、総合判断を返す。

## レビュー実行フロー

以下を **必ず順番通り** に実施する。

### Step 1: コンテキスト収集（メインエージェントが直接実施）

各ペルソナに渡す共通コンテキストを集める。

```bash
# 1. ブランチ・差分の状況確認
git status
git log main..HEAD --oneline
git diff main...HEAD --stat
git diff main...HEAD
```

加えて以下のドキュメントを **Read で読み込んでおく**（存在するもののみ）。サブエージェントには「これを読め」と指示するだけでなく、ファイルパスを明示して伝える。

- `CLAUDE.md`（プロジェクト規約）
- `docs/requirements.md` / `docs/requirements-definition.md` 等の要件定義書
- `docs/screen-definitions.md` / `docs/feature-list.md` 等の仕様書
- `docs/nonfunctional-requirements.md`
- `.claude/agents/code-reviewer.md`（共通レビュー観点・必ず読み込む）
- 変更対象ファイルの周辺コード（差分だけでは判断できない部分）

### Step 2: 5ペルソナのサブエージェントを並列起動

**1メッセージで5つの Agent ツールコールを並列実行する。** subagent_type は `general-purpose` を使用。

各エージェントには以下を **必ずプロンプトに含める**：

1. **ペルソナ宣言**（あなたは○○です）
2. **`.claude/agents/code-reviewer.md` の内容を全文埋め込む**（共通の観点・出力フォーマット・行動ルール）
3. **ペルソナ固有の重点観点**（後述）
4. **レビュー対象の差分**（`git diff main...HEAD` の出力をそのまま貼る、または「あなたが Bash で取得すること」と明示）
5. **読むべき仕様書・規約のファイルパス一覧**
6. **出力フォーマット**（code-reviewer.md の形式に従う）

### ペルソナ定義

#### Persona 1: シニアフロントエンドアーキテクト
**重点観点:**
- React コンポーネント設計（コンポーネント分割粒度、責務、Props 設計）
- カスタムフックの設計（依存配列、副作用、状態の局所化、useBooks/useFirstLaunch 等の既存規約）
- 状態管理の妥当性（Context / Zustand / 単純な useState の選択）
- shadcn/ui 規約（`app/lib/utils.ts` の `cn`、パステル OKLCH トークン、コンポーネント差し替え方針）
- TypeScript 型設計（any/unknown の濫用、ジェネリクスの妥当性、型の表現力）
- Vite/ビルド設定への影響、依存追加の妥当性
- 純粋関数はアロー関数で書く規約

#### Persona 2: QAエンジニア（テストスペシャリスト）
**重点観点:**
- 単体テスト（Vitest）が `.claude/skills/create-ut/SKILL.md` の方針に沿っているか
  - 古典派テスト・振る舞い駆動・テーブルテスト
  - 2階層 describe + 日本語観点 it
  - 観点重複の回避
- E2E テスト（Playwright）が `.claude/skills/create-e2e/SKILL.md` の方針に沿っているか
  - ユーザーストーリーベース・1テスト1シナリオ・各テスト独立
  - アクセシビリティロケータ優先（getByRole 等）
- 重要な振る舞いに対するテスト漏れ
- テストの信頼性（フレーキー要素・タイミング依存・実装詳細への結合）
- mock の濫用、過剰な内部結合検証
- 異常系・境界値の網羅

#### Persona 3: プロダクトオーナー（仕様整合性チェッカー）
**重点観点:**
- 要件定義書・ユーザーストーリーで定義された機能を実装しているか
- 仕様書に記載されていない挙動が混入していないか（スコープクリープ）
- 画面遷移・バリデーション・エラーメッセージが仕様通りか
- 子ども向けPWAとしてのUX妥当性（操作の単純さ・読み上げ可能なテキスト・ガチャ演出）
- CLAUDE.md の「プロジェクト概要」「対象ユーザー」と矛盾しないか
- コミットメッセージが Conventional Commits 規約に従っているか

#### Persona 4: セキュリティ＆アクセシビリティエンジニア
**重点観点:**
- OWASP Top 10（XSS・dangerouslySetInnerHTML の使用・入力検証）
- LocalStorage / IndexedDB に格納するデータの妥当性（機密情報を入れていないか）
- 外部APIへの呼び出しがある場合のCSP・CORS・トークン管理
- 子どもユーザーを想定したアクセシビリティ
  - WAI-ARIA 属性の適切な使用
  - キーボード操作・フォーカス管理
  - 読み上げ可能なラベル・alt 属性
  - 色のコントラスト比（パステルOKLCHトークン使用時の可読性）
- prefers-reduced-motion 配慮（ガチャ演出など）

#### Persona 5: コードリーダビリティ職人
**重点観点:**
- 命名（変数・関数・コンポーネント・ファイル）が意図を伝えているか
- マジックナンバー・マジックストリング
- YAGNI 違反（先回り実装・不要な抽象化）
- デッドコード・コメントアウトされた残骸
- 同一ロジックの重複（既存コードの再利用機会の見逃し）
- コメントの過不足（自明な what コメント／非自明な why の欠落）
- 関数長・引数数・ネストの深さ

### Step 3: 結果の統合・総合判断（メインエージェントが直接実施）

5つのエージェントから返ってきたレビュー結果を統合し、以下のフォーマットで出力する。

```markdown
# コードレビュー総合結果

## サマリー
- レビュー対象: `<branch>` ← `main`（コミット数: N、変更ファイル: M）
- 総合判定: ✅ マージ可 / ⚠️ 修正推奨 / 🔴 マージ不可
- 重大度別の指摘件数: 🔴 高 N件 / 🟡 中 N件 / 🟢 低 N件

## 総合判断
（各ペルソナの結論をふまえた1〜3文の判断）

## ペルソナ別レビュー要約

### 🏛️ シニアフロントエンドアーキテクト
（要点のみ。詳細指摘は下のセクションへ）

### 🧪 QAエンジニア
...

### 📋 プロダクトオーナー
...

### 🔐 セキュリティ＆アクセシビリティ
...

### ✍️ コードリーダビリティ職人
...

## 重大度別 指摘事項一覧

### 🔴 高（必ず修正）
- [ペルソナ名] **見出し**
  - 場所: `path/to/file.ts:42`
  - 問題: ...
  - 改善案: ...

### 🟡 中（修正推奨）
（同形式）

### 🟢 低（任意）
（同形式）

## ペルソナ間で意見が分かれた点
（あれば。例: アーキテクトは抽象化を推すが可読性職人は YAGNI 指摘）

## 良い点
（複数ペルソナが評価したポイントを集約）

## 次のアクション
（推奨される対応の優先順位）
```

#### 統合時のルール

- **重複指摘はマージ**（同じファイル・同じ問題は1件にまとめ、複数ペルソナ名を併記）
- **意見の対立は隠さない**（「ペルソナ間で意見が分かれた点」セクションに明記し、推奨判断を述べる）
- **総合判定の基準**:
  - 🔴 **マージ不可**: 重大度「高」が1件以上、または仕様との重大な不整合あり
  - ⚠️ **修正推奨**: 重大度「中」が3件以上、またはテスト不備が複数指摘
  - ✅ **マージ可**: 上記いずれにも該当しない（軽微な指摘のみ）

## 重要な制約

- **既存コードの問題は指摘しない**：今回の差分に含まれる変更のみが対象（code-reviewer.md の方針に準拠）
- **フォーマッタ・リンターで解決できる些細なスタイル問題は指摘しない**
- **指摘なしの場合は「指摘事項なし」と明示する**
- **サブエージェントの結果は信頼するが鵜呑みにしない**：明らかに間違った指摘（実在しないファイル・誤読）は統合時に除外する
- **5ペルソナは必ず並列で起動**：1メッセージ内で5つの Agent ツールコールを同時に発行する

## 発動条件

- `/code-review` 明示呼び出し
- 「コードレビューして」「ブランチレビューして」「PR前にレビュー」「マージ前チェック」などの自然言語
- 機能実装完了後に「レビューお願い」と言われたとき

## 対象外

- 要件定義書のレビュー → `requirements-reviewer` エージェントを使う
- 単一ファイルへの軽い助言 → 通常の応答で対応（このスキルはブランチ全体の差分が対象）
