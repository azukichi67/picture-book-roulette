# implement-code

えほんガチャプロジェクトでコード実装作業を行うときの標準ワークフロー。実装からテスト追加・レビュー・最終チェックまでを一貫した手順で進め、品質を担保した状態でユーザーに引き渡す。

## 前提

- このスキルは**コード実装作業専用**。要件定義・モック作成・既存テストの追加のみ・レビューのみといった単独タスクには使わない。
- 実装方針は事前の Plan モード等で合意済みであること。合意がない状態で着手しない。
- プロジェクトの規約は `CLAUDE.md` とユーザー auto memory（純粋関数はアロー関数 / UI スタックは shadcn/ui 等）に従う。

## ワークフロー

以下の手順を順番に実行する。途中で判断に迷うことがあれば**必ずユーザーに確認**してから進む。

### 1. 事前確認

1. ユーザーから受け取った指示・Plan モードで合意した計画を確認する。計画がない場合は実装着手前にユーザーへ確認する。
2. **worktree を使うかをユーザーに確認する**。issue 番号紐付きの実装は CLAUDE.md の方針通り原則 worktree + 新ブランチ（`<issue番号>-<概要>`）だが、毎回ユーザーに確認すること。
3. 関連コード・ファイルを Read / Grep で把握し、影響範囲を見積もる。既存コードの再利用余地を必ず探す。

### 2. 実装

1. 計画通りに実装する。計画にない変更は加えない（リファクタ・無関係な改善・将来用の抽象化を勝手に追加しない）。
2. 既存コードの再利用を優先し、重複を生まない。
3. ユーザー auto memory のルール（純粋関数はアロー関数 / 共通 UI は shadcn/ui / `cn` は `app/lib/utils.ts` 等）に従う。
4. 不明点が出たら推測せずユーザーに確認する。

### 3. テスト追加（スキップ可）

変更内容に応じてテストの種類を選ぶ。**スキップする場合は理由を明示してユーザーに伝えること**。

- **単体テスト**が必要なケース（hooks / services / utils / components のロジック変更）→ `create-ut` スキルを呼ぶ
- **E2E テスト**が必要なケース（ユーザーフロー・画面遷移・主要シナリオの追加変更）→ `create-e2e` スキルを呼ぶ
- どちらも該当する場合は両方
- スタイル微調整・型定義のみ等、テスト不要と判断した場合はスキップ可（理由を述べる）

### 4. コードレビュー（1ラウンド目）

1. `code-review` スキルを呼んでマルチペルソナレビューを実施する。
2. 返ってきた指摘を整理し、対応方針を決める：
   - **基本的に全件対応する**
   - 対応要否や対応方法に**判断が割れる指摘は必ずユーザーに確認**してから着手する（独断で「対応不要」と判断しない）
3. 指摘を反映する。テスト修正が必要なら create-ut / create-e2e で追加・修正する。

### 5. コードレビュー（2ラウンド目・最終）

1. もう一度だけ `code-review` スキルを呼ぶ。
2. 残った指摘を 4 と同じ方針（全件対応 / 迷ったらユーザー確認）で対応する。
3. **3 ラウンド目以降は回さない**。レビューはここで打ち止め。残課題があればユーザーに明示して判断を仰ぐ。

### 6. 最終チェック（NG ゼロを確認）

以下を**全部**実行し、NG（エラー・失敗）が一件もない状態にしてからユーザー報告に進む。NG が出たら必ず修正してから再実行する。

```bash
npm run lint       # Biome lint
npm run format     # Biome format（差分が出たら変更内容に含める）
npm run typecheck  # react-router typegen + tsc
npm test           # Vitest（単体テスト）
npm run test:e2e   # Playwright（E2E テスト）
```

注意点:

- `npm run format` で差分が出た場合、それも変更内容に含める（フォーマット差分が残った状態で完了報告しない）。
- `test:e2e` は時間がかかるため、Bash の timeout を十分長く設定する（必要に応じて `run_in_background` 検討）。
- ローカル環境特有のフレーキー失敗が疑われる場合は 1 度だけ再実行し、それでも落ちるなら原因を調査する。**フレーキーで片付けてはいけない**。

### 7. ユーザーへの完了報告

以下を簡潔に伝える（過剰な説明や手順の再掲は不要）：

- 実装した内容（1〜3 行サマリ）
- 追加・変更したテスト（または「テストなし」と理由）
- 最終チェックの結果（lint / format / typecheck / test / test:e2e すべて pass）
- 残課題があれば明示（例: レビュー 2 ラウンド目で対応見送りとなった指摘）
- PR 作成は**含まない**（必要なら別途 `create-pr` スキルを呼ぶようユーザーに案内）

## やらないこと

- PR 作成（`create-pr` スキルの担当）
- 要件定義（`requirements-definition` スキルの担当）
- モック作成（`mock-creator` スキルの担当）
- レビューのみの依頼への対応（`code-review` スキルを直接呼ぶ）
- 計画なしでの実装着手
- レビュー指摘の独断スキップ（迷ったらユーザー確認）
- フォーマット差分や lint NG が残ったままの完了報告
