# 非機能要件書生成スキル

## 出力構成

非機能要件書は以下の6カテゴリで構成する。

| カテゴリ | 定義 |
|----------|------|
| 可用性 | どのくらい止まらないか（稼働率、計画停止） |
| 性能 | どのくらい早いか（応答時間、同時接続数） |
| 拡張性 | 将来の規模拡大に対応できるか |
| 運用保守 | 運用しやすいか（ログ、監視、バックアップ） |
| 移行性 | データ移行・環境移行ができるか |
| セキュリティ | 安全か（認証、暗号化、アクセス制御） |

## 各カテゴリの出力フォーマット

各カテゴリをセクション見出しで区切り、以下の表を出力する。

| 要件ID | 要件名 | 内容 | 重要度 | 備考 |
|--------|--------|------|--------|------|

| 列 | 説明 | 形式・例 |
|----|------|---------|
| 要件ID | カテゴリ略称＋連番 | `AVL-001`（可用性）、`PER-001`（性能）、`SCA-001`（拡張性）、`OPS-001`（運用保守）、`MIG-001`（移行性）、`SEC-001`（セキュリティ） |
| 要件名 | 要件の名前（簡潔に） | `オフライン動作` |
| 内容 | **具体的・測定可能な言葉**で記述 | `インターネット接続なしで全機能が使用できること` |
| 重要度 | 高 / 中 / 低 | システムの特性に合わせてメリハリをつける |
| 備考 | 補足・根拠（なければ空欄） | |

## 作成ルール

1. **曖昧な言葉を使わない** — 「普通」「高速」は禁止。数値や条件で表現する
   - NG: `なるべく早く表示する` → OK: `画面遷移を1秒以内に完了する`
2. **「当たり前」も明文化する** — 暗黙の前提こそ書く
   - 例: `パスワードは暗号化して保存する`、`通信はHTTPSで行う`
3. **重要度にメリハリをつける** — 全項目を「高」にしない。システムの社会的影響度に応じて判断する
4. **記述のない要件は推測で書かない** — 要件定義書に根拠がない場合は備考に「要確認」と記載する
5. **移行性は不要なら省略する** — 新規開発で移行対象がない場合はカテゴリごと省略してよい

## 手順

1. 要件定義書（または指定されたドキュメント）を読み込む
2. 非機能要件の記述を抽出し、6カテゴリに分類する
3. 各カテゴリの表を上記フォーマットで出力する（ドラフト）
4. 要件定義書に記載がないカテゴリでも、システムの特性から明らかに必要な「当たり前」要件を補完する（備考に「要件定義書に記載なし・補完」と明記）
5. **必ず** `requirements-reviewer` サブエージェントを呼び出し、ドラフトをレビューさせる
6. レビューで指摘された 🔴高・🟡中 の問題をすべて修正する
7. 修正済みの成果物を利用者に返す。末尾に「機能要件・スコープ外は除外した」旨と、レビューで修正した主な点を一言添える

> **注意:** ステップ5を省略してはならない。レビューなしで成果物を返すことは禁止。
