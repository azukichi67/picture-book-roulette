# 要件定義スキル

## 概要

このスキルはシステム・アプリの要件定義を、ヒアリングから要件定義書の完成まで一貫して支援する。
詳細な知識ベースは [references/requirements-knowledge.md](references/requirements-knowledge.md) を参照すること。

## ワークフロー

### Phase 1: 現状把握（ヒアリング）

まず以下を確認する。**一度に全部聞かず、2〜3問ずつ対話形式で進めること。**

**プロジェクト概要**
- 何を作りたいか（システム・機能の概要）
- なぜ作るか（課題・背景）
- いつまでに必要か（スケジュール感）

**ステークホルダー**
- 誰が使うか（エンドユーザー）
- 誰が決裁するか
- 関係する部署・役割

**現状業務**
- 今はどうやっているか（As-Is）
- 何が一番困っているか
- 改善後の理想の姿（To-Be）

> **ヒアリングの鉄則**
> - 「何が欲しいですか？」より「何が一番困っていますか？」
> - 「なぜ？」を繰り返して本当のニーズを掘り下げる
> - 「〇〇みたいに」という表現は必ず具体化する
> - 例外ケースを積極的に聞き出す（「〜の場合はどうしますか？」）

### Phase 2: 要件の整理

ヒアリング内容をもとに以下を整理する。

**機能要件**
- 機能一覧（機能ID、機能名、機能概要、優先度、関連画面、備考）
- 複雑な機能はユースケース（アクター、事前条件、基本フロー、例外フロー）で詳細化
- 画面一覧（画面ID、画面名、概要、アクセス権限）

**非機能要件**
6大カテゴリ（可用性、性能、拡張性、運用保守、移行性、セキュリティ）を漏れなく確認する。
「当たり前」も明文化する（「パスワードを暗号化する」等）。

> **要件記述の鉄則**
> - 曖昧な表現を排除する：「使いやすく」→「3クリック以内」
> - すべての要件を検証可能な形にする
> - 「あとで決める」を極力排除する

### Phase 3: 要件定義書の作成

以下の構成で要件定義書（Markdown）を作成する。

```
1. エグゼクティブサマリー（全体像を1ページ相当で）
2. プロジェクト概要（目的、背景、スコープ、スケジュール）
3. ステークホルダー一覧
4. 業務フロー（As-Is / To-Be）
5. 機能要件
   5.1 機能一覧表
   5.2 ユースケース（複雑な機能のみ）
   5.3 画面一覧
   5.4 帳票一覧（あれば）
6. 非機能要件
7. 制約条件・前提条件
8. 未決事項（決まっていない項目はリスクとともに明記）
9. 用語集
```

### Phase 4: requirements-reviewer によるレビュー（必須）

**要件定義書を作成したら、必ず以下を実行すること。利用者への返却前にこのフェーズを省略してはならない。**

1. `requirements-reviewer` エージェントに要件定義書ファイルを渡してレビューを依頼する
2. 指摘事項（曖昧さ、漏れ、矛盾、検証不可能な記述、ユーザーストーリーとの不整合）を確認する
3. すべての指摘事項を要件定義書に反映する
4. 反映後、再度レビューが必要な重大な指摘がある場合は再レビューを実施する

### Phase 5: 利用者への返却

- 完成した要件定義書の内容を提示する
- レビューで指摘された主な点と、どう対応したかを簡潔に報告する
- 利用者に確認・フィードバックを求める

## 出力ファイル

要件定義書は `docs/requirements.md` に保存する（プロジェクトの `docs/` ディレクトリが存在する場合）。
存在しない場合は利用者に保存先を確認する。
