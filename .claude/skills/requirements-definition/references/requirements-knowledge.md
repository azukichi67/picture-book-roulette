# 要件定義の知識ベース

## 要求 vs 要件

| | 要求 | 要件 |
|---|---|---|
| 定義 | 顧客の希望・願望 | システムが満たすべき条件 |
| 特徴 | 曖昧、主観的 | 具体的、検証可能 |
| 例 | 使いやすくしたい | 3クリック以内で目的の画面に到達できる |
| 例 | 早くしたい | 検索結果が2秒以内に表示される |

**鉄則：要件は数値や状態で検証可能な形に落とす**

## ステークホルダー整理

| ステークホルダー | 関心事 | よくある要望 |
|---|---|---|
| 経営層 | コスト、ROI、競争力 | 早く安く、戦略に沿ったもの |
| 現場ユーザー | 使いやすさ、業務効率 | 今の業務を楽にしたい |
| IT部門 | 保守性、セキュリティ | 運用しやすいもの |
| 顧客 | サービス品質 | 便利で早いサービス |

ステークホルダーマップの2軸：影響力 × 関心度
- 影響力高・関心度高 → 密に連携
- 影響力高・関心度低 → 満足させる（定期報告）
- 影響力低・関心度高 → 情報提供
- 影響力低・関心度低 → 監視（必要時のみ）

**注意：声の大きい人＝正しい人ではない。現場の声なき声も拾う**

## ヒアリングの技術

### 良い質問 vs 悪い質問

| 悪い質問 | 問題点 | 良い質問 |
|---|---|---|
| どんな機能が欲しいですか？ | 相手が答えを持っていない | 今、一番時間がかかっている業務は何ですか？ |
| 使いやすくしたいですか？ | Yes以外の答えがない | 今のシステムで困っていることを3つ挙げるとしたら？ |

### なぜの深掘り
「なぜ？」を繰り返して本当のニーズを探る。
例：「Excel出力したい」→なぜ？→「加工したい」→なぜ？→「上司に報告するため」→なぜ加工？→「必要な情報が足りないから」
→ 本当の要件：「報告に必要な情報を表示する」

## 業務フロー

### As-Is → To-Be
1. まず現状（As-Is）を描く：現状を知らないと何を変えるべきかわからない
2. その差分がシステム化のスコープ

### 例外ケースを引き出す質問
- 「承認者が休みの時はどうしますか？」→ 代理承認フロー
- 「急ぎの場合はどうしますか？」→ 緊急時の特例
- 「間違えて登録した場合は？」→ 取消・修正フロー
- 「却下された後はどうなりますか？」→ 再申請フロー
- 「年度末など特殊な時期は？」→ 期間限定の特殊ルール

**例外は「滅多に起きない」が「起きたら致命的」**

## 機能要件の書き方

### 機能一覧表の項目

| 項目 | 説明 | 例 |
|---|---|---|
| 機能ID | 一意の識別子 | F-001 |
| 機能名 | 機能の名前 | ユーザー登録 |
| 機能概要 | 何ができるか | 新規ユーザーの情報を登録する |
| 優先度 | 必須か任意か | 必須 / 任意 |
| 関連画面 | どの画面で使うか | ユーザー登録画面 |
| 備考 | 補足情報 | 管理者のみ実行可能 |

### ユースケースの構成要素
- アクター：誰が使うか
- 事前条件：この機能を使う前の状態
- 基本フロー：正常な流れ
- 代替フロー：分岐する場合
- 例外フロー：エラーの場合
- 事後条件：この機能を使った後の状態

### 曖昧な表現 → 具体的な表現

| 悪い例 | 良い例 |
|---|---|
| 使いやすくする | 3クリック以内で完了 |
| 早くする | 応答時間2秒以内 |
| たくさん保存できる | 10万件まで保存可能 |

**「曖昧な言葉は、後で揉める言葉」**

## 非機能要件の6大カテゴリ

| カテゴリ | 内容 | 例 |
|---|---|---|
| 可用性 | どのくらい止まらないか | 稼働率99.9%、計画停止は月1回まで |
| 性能 | どのくらい早いか | 応答時間2秒以内、同時接続100人 |
| 拡張性 | 将来の拡張に対応できるか | ユーザー数10倍でも対応可 |
| 運用保守 | 運用しやすいか | ログ出力、監視機能、バックアップ |
| 移行性 | データ移行できるか | 旧システムから3年分のデータ移行 |
| セキュリティ | 安全か | 通信暗号化、アクセス制御、監査ログ |

「当たり前」も明文化する：
- 日本語で表示される
- ブラウザの戻るボタンが使える
- パスワードは暗号化して保存する

## 要件定義書の標準構成

1. エグゼクティブサマリー（1ページで全体像）
2. プロジェクト概要（目的、背景、スコープ）
3. ステークホルダー一覧
4. 業務フロー図（As-Is / To-Be）
5. 機能要件（機能一覧、ユースケース）
6. 画面一覧・画面モック
7. 帳票一覧
8. 非機能要件
9. 制約条件・前提条件
10. 用語集

## よくある罠

### 「あとで決めましょう」の危険性
- 開発開始後は決める余裕がない
- 決まらず開発者が勝手に決める
- 対処：選択肢を3つ提示して選んでもらう。期限も切る

### スコープクリープ
- 「ついでにこれも」の積み重ねで工数・予算オーバー
- 追加要件は必ず影響範囲を調査してから対応方針を決める

### 「〇〇みたいに」の危険性
- 「Excelみたいに」→ 一覧編集？計算？コピペ？
- 「LINEみたいに」→ リアルタイム？既読？スタンプ？
- 必ず「具体的には？」と聞く
