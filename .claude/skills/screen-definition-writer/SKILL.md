# 画面定義書作成スキル

## ワークフロー

### Step 1: インプット確認

`docs/requirements.md` を読み込む。存在しない場合は利用者にファイルパスを確認する。

以下を抽出する：
- 画面一覧（画面ID・画面名）
- 各画面の機能要件・ユースケース
- 画面遷移
- データモデル（入力項目・型の参照に使用）

### Step 2: 画面定義書の作成

画面一覧に含まれる**すべての画面**について、`references/screen-template.md` のフォーマットに従って記述する。

**記述の原則：**
- 曖昧表現を排除し、検証可能な仕様を書く（「適切に」「わかりやすく」は禁止）
- バリデーションは条件・エラーメッセージを具体的に記述する
- ボタン操作は「何をして・どこへ遷移するか」を明記する
- 要件定義書に記載がない仕様を推測で補わない。不明な場合は `[要確認]` と記載する

### Step 3: ファイル出力

`docs/screen-definitions.md` に保存する（`docs/` が存在する場合）。存在しない場合は利用者に確認する。

### Step 4: requirements-reviewer によるレビュー（必須）

**成果物を利用者に返す前に必ずこのフェーズを実施すること。省略禁止。**

1. `requirements-reviewer` エージェントに作成した `docs/screen-definitions.md` を渡してレビューを依頼する
2. 指摘事項（曖昧さ・漏れ・矛盾・検証不可能な記述）を確認する
3. すべての指摘事項を反映する

### Step 5: 利用者への返却

- 完成した画面定義書の内容を提示する
- レビューで指摘された主な点と対応内容を簡潔に報告する
- 利用者に確認・フィードバックを求める
